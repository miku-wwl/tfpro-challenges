# Terraform Challenge 教程编写指南

本指南用于统一编写 Challenge 1～8 的解题教程。目标是：**步骤清楚、命令可执行、方便考试时快速查官网**，不写成长篇原理文章。

## 一、文件约定

教程统一放在仓库主目录：

```text
challenge-1-tutorial.md
challenge-2-tutorial.md
...
challenge-8-tutorial.md
```

每篇教程只针对对应 Challenge，不修改原题目的 `Readme.md`。

## 二、编写前必须先实际解题

Agent 不能只阅读题目后直接推测答案。编写教程前，必须先把对应 Challenge **完整手动解决并验证一遍**，再根据实际过程整理教程。

要求如下：

1. 先检查题目目录、现有代码、变量文件和任务间的依赖关系。
2. 按题目顺序实际修改代码并执行相关 Terraform 命令。
3. 记录遇到的报错、资源地址、命令转义和最终有效的解决方法。
4. 每个任务都通过相应验证；涉及重构或 import 时，必须检查 `terraform plan` 是否符合题目要求。
5. 确认完整流程能够走通后，才开始写教程。教程中只保留最终正确且必要的步骤，不记录无关的试错过程。
6. 实际解题时保留题目原始说明，不覆盖用户已有修改。AWS 资源默认优先使用本地 Docker + LocalStack 完成创建、导入、state 和销毁实验，避免影响真实云资源。
7. LocalStack 不支持或行为与真实 AWS 不一致的功能，需要明确记录差异；只有在已获得相应环境和授权后才能改用真实 AWS。无法实际执行的部分必须在教程中明确标注“仅完成静态验证”，不能声称已验证成功。

### LocalStack 验证约定

开始解题前先确认容器和服务状态：

```bash
docker ps
curl http://localhost:4566/_localstack/health
```

Terraform 的 LocalStack 实验配置应与考试答案分开保存，不能把 endpoint、测试凭证或跳过校验的参数混入最终考试代码。实验时可使用单独的 `localstack_override.tf`：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    iam = "http://localhost:4566"
    s3  = "http://localhost:4566"
    ec2 = "http://localhost:4566"
    sts = "http://localhost:4566"
  }
}
```

如果原题已经包含默认的 `provider "aws"`，实验副本中应修改原 provider，不能再添加一个重复的默认 provider。教程中的正式答案仍使用题目要求的标准 AWS provider 配置。

LocalStack 实测至少应覆盖题目涉及的以下流程：

```bash
terraform init
terraform validate
terraform apply
terraform state list
terraform plan
terraform destroy
```

涉及 import、`state mv`、`state rm` 或 backend 时，也必须在 LocalStack 中实际执行对应操作。教程开头的“验证说明”需要写明 LocalStack 版本、Terraform 版本、AWS Provider 版本，以及未覆盖的真实 AWS 差异。

## 三、每篇教程的固定结构

```markdown
# Challenge N 解题教程

## 验证说明
说明本教程实际验证过的 Terraform/AWS 环境，以及未能实际执行的部分。

## 题目目标
用 2～4 行说明考查内容和最终结果。

## 开始前检查
- 当前工作目录
- AWS 身份或 Profile
- Terraform 初始化状态
- 题目要求保留、导入或禁止销毁的资源

## Task 1：题目中的任务名

### 解题思路
简要说明要改什么、为什么，最多 3～5 条。

### 操作步骤
给出文件位置、必要代码和命令，按执行顺序排列。

### 验证
给出 `terraform fmt`、`terraform validate`、`terraform plan`、
`terraform state list` 或输出检查方式，并说明预期结果。

### 难点与官网入口
- 搜索关键词：`Terraform ...`
- 官网：[页面名称](官网 URL)

## Task 2：……
重复以上结构。

## 最终检查
列出交卷前必须确认的结果。
```

简单任务不必强行保留全部小标题，可以合并成“操作与验证”。只有真正的难点才添加官网入口。

## 四、写解题步骤的规则

1. **严格按题目顺序写。** 后续任务依赖前一步状态时，要明确提醒，不把所有代码一次性贴出来。
2. **先说文件，再给修改。** 例如：“在 `outputs.tf` 中添加”，不要只给一段来历不明的代码。
3. **代码只保留关键部分。** 完整文件很短时可以给完整代码；否则只展示新增或修改的 block。
4. **命令必须可复制。** 标明应在哪个目录执行；资源地址含 `count` 或 `for_each` 索引时要加引号，并提醒 PowerShell 与 Bash 的转义差异。
5. **变量值不写死。** 账号、ARN、资源 ID、随机名称等用 `<ACCOUNT_ID>`、`<BUCKET_NAME>` 表示，并说明如何从 output、state 或 AWS CLI 查到。
6. **危险操作必须提前警告。** 删除 state、`state rm`、import、backend 迁移、destroy 前，先写备份或 `plan` 检查步骤。
7. **以零变更作为重构验收。** 模块移动、import、data source 替换等任务，最终应尽量得到：

   ```text
   Plan: 0 to add, 0 to change, 0 to destroy.
   ```

8. **不要替题目增加架构。** 除非题目要求，不引入额外模块、复杂 locals、脚本或第三方工具。

## 五、各类题目的写法

| 题目类型 | 教程必须说明 |
| --- | --- |
| 修复错误 | 错误位置、原因、最小修改、`validate/plan` 结果 |
| `count` / `for_each` | 实例地址形式、`count.index` 或 `each.key/value` 的来源 |
| CSV / 表达式 | `file` → `csvdecode` → 过滤/转换 → 最终数据类型 |
| Output | 输出值表达式，以及如何用 `terraform output` 验证或写入文件 |
| 模块重构 | 模块目录、输入、输出、调用关系，以及对应的 `state mv` |
| Import | 配置中的目标地址、云端资源 ID、import 命令、import 后的零变更检查 |
| State 操作 | 操作前备份、完整资源地址、操作后 `state list` 和 `plan` |
| Backend | backend 配置、初始化/迁移命令、远端 state 的验证方法 |
| 多 Provider/Profile | 凭证来源、profile/role 链、provider alias、模块中的 providers 映射 |
| 生命周期 | `lifecycle` 解决的问题，以及修改前后的 plan 对比 |
| Destroy | 销毁范围、执行顺序、遗留资源检查；题目要求保留的对象不能误删 |

## 六、官网链接的选择规则

官网链接应紧跟在相关难点后，不要统一堆在文末。每个难点通常给 1 个主链接，最多 2 个。

优先级如下：

1. Terraform 语言或 CLI：使用 `developer.hashicorp.com/terraform`。
2. AWS 资源参数和 import ID：使用 Terraform Registry 中对应的 AWS Provider 资源页。
3. AWS Profile、AssumeRole 等行为：Terraform 文档不足时，再链接 AWS 官方文档。

常用入口：

- [`count`](https://developer.hashicorp.com/terraform/language/meta-arguments/count)
- [`for_each`](https://developer.hashicorp.com/terraform/language/meta-arguments/for_each)
- [For expressions](https://developer.hashicorp.com/terraform/language/expressions/for)
- [Functions](https://developer.hashicorp.com/terraform/language/functions)
- [Output values](https://developer.hashicorp.com/terraform/language/values/outputs)
- [Modules](https://developer.hashicorp.com/terraform/language/modules)
- [`terraform state mv`](https://developer.hashicorp.com/terraform/cli/commands/state/mv)
- [`terraform state rm`](https://developer.hashicorp.com/terraform/cli/commands/state/rm)
- [Import resources](https://developer.hashicorp.com/terraform/language/import)
- [S3 backend](https://developer.hashicorp.com/terraform/language/backend/s3)
- [Provider configuration](https://developer.hashicorp.com/terraform/language/providers/configuration)
- [`lifecycle`](https://developer.hashicorp.com/terraform/language/meta-arguments/lifecycle)
- [AWS Provider 文档](https://registry.terraform.io/providers/hashicorp/aws/latest/docs)

考试时的搜索写法应在教程中直接给出，例如：

```text
Terraform for expression filtering
Terraform state mv module
Terraform import for_each
Terraform AWS aws_s3_object import
Terraform S3 backend configuration
```

## 七、每个任务的验证顺序

根据任务选择执行，不要求每一步全部运行：

```bash
terraform fmt -recursive
terraform init
terraform validate
terraform plan
terraform state list
terraform output
```

涉及 state 或 import 时，额外检查：

```bash
terraform state show '<RESOURCE_ADDRESS>'
terraform plan
```

教程不能只写“运行成功”，必须给出可判断的预期结果，例如：无语法错误、资源数量正确、指定地址出现在 state 中，或 plan 为零变更。

## 八、篇幅控制

- 开头不重复粘贴完整题目，只概括目标。
- 一个 Task 通常控制在 5～15 行说明，加必要代码。
- 基础命令不解释 Terraform 原理。
- 同一知识点后续再次出现时引用前文，避免重复。
- 每篇末尾只保留一份最终检查清单，不写总结性套话。
